Chapter 2 · Large Language Models

How the "brain" actually works 🧠

From An Illustrated Guide to AI Agents — Jay Alammar & Maarten Grootendorst (O'Reilly, Early Release)

The one-line version: An LLM is a machine that reads text as little chunks (tokens) and predicts the next chunk, one at a time. This chapter opens the hood: how it's built, how it's trained, and what's inside it (the Transformer).
How to read this The book splits this chapter into Part 1 (the must-know basics — that's most of these notes) and Part 2 (an optional deep dive into the maths). For your interview, Part 1 is the priority. I've clearly marked where Part 2 begins so you can stop there if your head is full. 🙂
PART 1What you should know about LLMs

1 The LLM's job inside an agent

  • The LLM sits in the middle, acting as a bridge between the user and the environment.
  • It does four jobs: read the user's question → choose an action → read the result that comes back → reply to the user.

2 Tokens in, tokens out

  • LLMs don't see whole words — they see tokens: little chunks that can be a word, part of a word, a number, or punctuation.
  • Example: the book shows flamingos becoming two tokens → flamingo + s (the exact split depends on the tokenizer). Chopping words into chunks lets the model handle words it's never seen.
  • It generates the answer one token at a time: predict a token → stick it onto the end → read the whole thing again → predict the next. This self-feeding loop is called autoregression.
🧱
Analogy: Tokens are like LEGO bricks for language — the model builds every sentence brick-by-brick, checking the whole tower so far before adding the next brick.

3 The 3 formats that turn an LLM into an agent

A raw LLM just continues text. To make it behave like a helpful, tool-using assistant, we wrap its input in special formats — ones the model is specifically trained (in post-training, §4) to understand and produce:

System prompt

  • A privileged instruction placed before the user's message that shapes behaviour and takes precedence over what users ask.
  • Set by whoever deploys the model (e.g. a company), not baked in during training — so you can change behaviour without retraining.
📋
Analogy: The system prompt is the employee handbook / standing orders pinned above the desk — the rules the assistant follows no matter who walks up.

Multi-turn conversation

  • Formats the chat as user ↔ assistant turns so the model can see who said what across a longer conversation. (A first taste of "memory" — proper memory is Ch.4.)

Tool-use

  • Available tools are usually listed in the system prompt. Instead of answering, the LLM can emit a structured tool call like web_search("Toronto Zoo flamingos").
  • The agent wrapper spots that pattern, runs the real tool, and feeds the result back in. (Full detail = Ch.5.)
  • How it tells them apart: the system / user / assistant roles & tool calls are marked by special tokens in a fixed chat format (e.g. OpenAI's Harmony) — that's how the model knows your question from its own instructions.

4 How an LLM is trained (2 phases)

FIRST — WHAT'S A "WEIGHT"?

An LLM is a giant neural network. Its know-how lives in millions-to-billions of numbers called weights (aka parameters) — think tiny dials. Training = tweaking those dials so its guesses improve. The "Large" in LLM literally means how many of them there are.

Phase 1 — Pre-training (language modelling)

  • Take mountains of text (web, books, code), hide the next word, make the model guess it.
  • Wrong guess → measure the error (the loss) → nudge the model's weights to do better. Repeat billions of times.
  • Result = a base model (the landmark one was OpenAI's GPT-3, 2020). It's knowledgeable but hard to steer.
📚
Analogy: A lifetime of fill-in-the-blank exercises over the whole internet. Get enough of them right and you've absorbed grammar, facts, code and reasoning patterns.

Phase 2 — Post-training (making it useful)

The base model has raw talent but needs polishing. Two steps:

  • ① SFT (Supervised Fine-Tuning), aka instruction-tuning: show it example prompt → ideal answer pairs so it learns to follow instructions. Neat detail: the prompt is fed in as context but only the answer part is graded during training.
  • Why not just SFT? SFT gets the model following instructions, but can't capture what makes an answer genuinely good (helpful, accurate, well-reasoned). That's RL's job.
  • ② Reinforcement Learning (RL): instead of showing the perfect answer, we score the answers it produces (a reward) and push it toward higher scores. Two flavours 👇
  • RLHF (from Human Feedback): humans (or a "reward model" trained on their preferences) mark one answer as preferred, another rejected. Model shifts toward the preferred style.
  • RLVR (with Verifiable Rewards): a program checks the answer automatically — did it use the right format? Is the maths correct? Scales beautifully where there's a clear right answer.

GRPO — a popular RLVR recipe

Full name: Group Relative Policy Optimization.

  • Ask the model the same question and let it write a whole group of answers at once (e.g. 5), using a bit of randomness so they differ.
  • Score each: e.g. format reward 0.3 (used the right tags?) + accuracy reward 0.7 (answer correct?).
  • Reward is judged relative to the group — reinforce the answers that beat their siblings. (That's the "Group" in GRPO. Used to train DeepSeek-R1 — Ch.3.)

5 Inside the Transformer (the anatomy)

Nearly all LLMs since ~2020 use the Transformer decoder. It has three big parts:

  • Tokenizer — chops text into tokens. Each token has an ID from a fixed vocabulary (say ~50,000), and each ID maps to an embedding = a list of numbers the model can compute with. The vocabulary also holds special tokens (like <|start|>) that mark the chat format from §3.
  • Transformer blocks — a tall stack; nearly all the actual "thinking" happens here (Section 6).
  • LM head — turns the final result into a probability score for every token in the vocabulary. High score = likely next token.

Picking the next token: temperature

  • The rule for choosing from those probabilities is the decoding strategy.
  • Temperature 0 = always pick the single highest-probability token (safe, repetitive).
  • Higher temperature = sampling: lower-probability tokens get a real chance (more varied/creative).
🎲
Analogy: A weighted dice roll. Temperature 0 = always play your strongest tile; crank it up = you gamble on riskier, more surprising words.

Two ideas every agent builder must know

  • Context length = how many tokens can flow through at once (the model's desk size / working memory). Everything — system prompt + history + question + answer — must fit. Choosing what to keep in that limited space is context engineering (recurring theme of the book).
  • Under the hood: all tokens flow through the blocks in parallel "tracks," but only the last token's output is used to predict the next word. The earlier tokens' results don't change — which is exactly why they can be reused 👇 (and the number of tracks = the context size).
  • KV-cache (aka prompt / prefix caching) = after the first pass, the model saves its work so each new token only needs a tiny bit of fresh computation. Huge speed & cost win — a key lever for agent developers (more in Ch.11).

6 Inside one block: memory + attention

Every Transformer block has two parts working together:

The feed-forward network = the knowledge store

  • A big neural network that does the heavy lifting of prediction and appears to store facts learned in training.
  • Give it "The Shawshank" → it predicts "Redemption", because that's the most common continuation it saw.
  • Model-picking heads-up: this network comes in two styles — a single dense network, or a Mixture-of-Experts (many small ones). It's a real choice when picking an open model — mechanics in Part 2 §9.

Self-attention = understanding context

  • Word order and context carry meaning: bank = money-place or river-side? You only know from context.
  • Self-attention lets each token look back at the most relevant earlier tokens to figure out what's going on — e.g. resolving what "it" refers to.
  • It works in two steps: ① score how relevant each earlier token is, then ② blend their info into the current token, weighted by those scores.
🔗
Analogy: It's pronoun detective work — to understand "it," you glance back at the sentence and connect it to the noun that fits.
If you stop here, you're interview-ready Tokens → predict next → trained by pre-training + SFT + RL → built from a tokenizer + Transformer blocks (attention + feed-forward) + LM head. Part 2 below is bonus depth.
PART 2Optional deep dive — the machinery

7 How self-attention is actually computed: Q, K, V

  • Each token's vector is multiplied by three learned matrices to make three new vectors: Query, Key, Value.
  • ① Relevance scoring: multiply the current token's Query by every token's Key → a relevance score each, then softmax (turns the scores into percentages that add to 100%; the √dk bit is just a scaling constant you can ignore for the intuition).
  • ② Combine: take a weighted sum of the Value vectors, weighted by those scores → the enriched output for the current token.
🔎
Analogy: Like a search engine. Your Query ("what am I looking for?") is compared against every document's Key ("what am I about?"); the best matches hand over their Value ("here's my info").

8 KV-cache & making attention cheaper

  • Problem: plain ("vanilla") attention recomputes the Keys & Values for every earlier token at every step — hugely wasteful as text grows.
  • Fix = KV-cache: store the K & V of past tokens and reuse them; only compute K & V for the new token. (Queries aren't cached — you only need the newest token's Query.)
  • Downside: the cache eats memory. So researchers invented leaner attention variants:
  • GQA / MQA (Grouped-Query / Multi-Query): share K & V across attention heads → smaller cache, a little less accurate.
  • MLA (Multi-head Latent Attention, DeepSeek): cache a small compressed version of K & V, expand it back when needed — leaner and more efficient than GQA, while (being basically full Multi-Head Attention) keeping accuracy high.
  • DSA (DeepSeek Sparse Attention): a lightning indexer + top-k selector attend to only the most relevant ~couple-thousand tokens instead of all of them.

9 Mixture of Experts (MoE)

  • Swap the one big feed-forward network for many smaller "expert" networks + a router.
  • Dense model = every part runs for every token. Sparse (MoE) model = the router sends each token to just a few experts.
  • Experts seem to specialise in fine-grained things (e.g. numbers vs verbs) — though exactly what they learn is still fuzzy.
  • Load balancing keeps one expert from hogging all the work — via expert capacity (a per-expert limit), auxiliary loss, or adding a little noise to the router.
  • Sparse vs active parameters: all experts are loaded in memory (sparse), but only a few run per token (active). That's why you see names like Qwen3-30B-A3B = 30B loaded, 3B active → big-model quality, small-model speed.
  • Example: DeepSeek-R1 has 256 experts per layer, picks 8, plus 1 always-on "shared" expert for general knowledge.
🏥
Analogy: A hospital keeps many specialists on payroll (loaded in memory), but each patient only sees the two or three they actually need (active). Cheaper per visit, same expertise available.
✦ the whole chapter on one line ✦
Recap: Text → tokens → the model predicts the next one. It's pre-trained (fill-in-the-blank on the internet) then post-trained (SFT + RL/RLHF/RLVR) to be helpful. Inside sits a Transformer: tokenizer → blocks of attention (context) + feed-forward (knowledge) → LM head → next token. Efficiency tricks: KV-cache, leaner attention (MLA/DSA), and Mixture-of-Experts.

★ Interview cheat-sheet 🎤

Say-it-out-loud answers to the questions most likely to come from this chapter.

Q. In one sentence, what is an LLM?
A model that breaks text into tokens and repeatedly predicts the next token — that simple loop, at massive scale, produces everything from essays to code.
Q. What's the difference between pre-training and post-training?
Pre-training = next-word prediction over huge text → a knowledgeable but hard-to-steer base model. Post-training (SFT + reinforcement learning) makes it follow instructions and match human preferences.
Q. RLHF vs RLVR?
RLHF scores answers using human preferences (preferred vs rejected). RLVR scores them with an automatic verifier (right format? correct answer?) — ideal for maths and code where correctness is objective.
Q. What are the parts of a Transformer?
A tokenizer (text→tokens), a stack of Transformer blocks (each = self-attention + feed-forward), and an LM head that outputs a probability for every possible next token.
Q. What does self-attention do?
Lets each token look back at the most relevant earlier tokens to understand context — like working out what "it" refers to. Computed with Query, Key, Value vectors.
Q. What is temperature?
A dial on how the next token is chosen. 0 = always pick the most likely word (predictable); higher = sample more surprising words (creative).
Q. Why do context length and the KV-cache matter?
Context length caps how much text fits in the model at once (so you must curate what you feed it — "context engineering"). The KV-cache reuses past computation to make generation faster and cheaper — critical for real agents.
Q. What is a Mixture-of-Experts model?
Instead of one big network, many small "experts" plus a router that activates only a few per token. You get a large model's quality at a small model's running cost (e.g. "30B loaded, 3B active").
